v1.5.9 - 23 Mar 2026
---------------------
CHANGED
- Removed "UNSURE" suffix from proposed filenames. Low-confidence files are

v1.5.8 - 23 Mar 2026
---------------------
ADDED
- New "Final Report" document type for loss adjuster final reports (e.g.
  Sedgwick Final Report). Previously these were misclassified as Claims Team
  FDL because "claim decision" appeared in the report text.

FIXED
- Sedgwick Final Reports no longer misclassified as Claims Team FDL. Added
  conflict resolution: Claims Team FDL loses to Final Report when both match.
- ClaimsCo-authored Notice of Response to AFCA now correctly identified:
  WHO=Complainant, ENTITY=ClaimsCo, WHAT="Notice of Response". Previously
  showed as FF / Notice of Response from QBE because QBE was mentioned in text.
- Added "the financial firm's" as ClaimsCo authorship detection phrase (used
  in complainant-side AFCA submissions referring to the insurer in third person).
- "Information Request Response" documents now classified as Request for
  Information instead of PDS.
- Date engine now skips "Date of Loss", "Date of Incident" and similar event
  date labels when inferring document dates for letters and reports. Prevents
  picking up the loss date (e.g. 24.12.2023) instead of the actual letter date.
- Date engine rejects pre-2000 dates for letter/report types and scans deeper
  into the document for the correct date (e.g. Morse Building report now picks
  up 26.07.2024 from page 2 instead of 07.12.1969 construction date).
- AFCA-authored documents now always show ENTITY=AFCA.
- Certificate of Insurance no longer overrides IDR FDL, Notice of Response,
  or Request for Information classifications.
- Delegation of Authority no longer overrides AFCA Submission classification.
- Added "complaint lodgement" and "lodgement of a formal complaint" keywords
  for AFCA Submission detection.


v1.5.7 - 19 Mar 2026
---------------------
FIXED
- Duplicate detection now catches more cases:
  - Field-based comparison now includes already-flagged records so a file
    with the same WHO+DATE+ENTITY+WHAT as an exact duplicate also gets caught
    (previously, exact duplicates were excluded from field comparison, allowing
    a third copy with different binary content to slip through unflagged).
  - One-per-claim document type checks similarly improved.
  - Rows whose fields literally contain "DUPLICATE" (from previous rename
    sessions) are now automatically flagged as Likely Duplicate.


v1.5.6 - 19 Mar 2026

  multiple approved files produce the same proposed name, the app now
  automatically appends (2), (3), etc. to resolve collisions instead of
  showing a "Duplicate target name" error.


v1.5.5 - 19 Mar 2026

v1.5.4 - 19 Mar 2026
---------------------
FIXED
- Manually editing a field (WHO, DATE, ENTITY, WHAT) on a duplicate-flagged
  row now correctly rebuilds the proposed filename from the edited values.
  Previously the filename stayed as "DUPLICATE.pdf" even after editing because
  the duplicate status was never cleared on manual edits. This also caused
  validation errors ("Duplicate target name") when trying to rename.
- Same fix applied to bulk field edits (Set WHO/Entity/WHAT/Date buttons)
  and Find/Replace operations.


v1.5.3 - 18 Mar 2026
---------------------
FIXED
- Root cause fix for entity detection: added Unicode NFKC normalisation to
  decompose PDF ligature characters (ﬁ→fi, ﬂ→fl, ﬀ→ff, etc.). PDFs
  frequently encode "fi" as the single ligature glyph U+FB01, causing
  "Certified" to be stored as "Certi\uFB01ed" which silently broke all
  substring matching against entity names. NFKC normalisation decomposes
  these back to standard characters.
- Changed PyMuPDF text extraction flags to clear TEXT_PRESERVE_LIGATURES,
  so ligatures are decomposed at extraction time as well (defence in depth).
- Added word-level text extraction fallback: if standard text extraction
  and deep scan both fail to find an entity, a word-by-word extraction pass
  captures text that may be missed due to unusual font encodings or PDF
  structure.

v1.5.2 - 18 Mar 2026
---------------------
FIXED
- PDF text cleaning now strips non-breaking spaces (U+00A0), zero-width
  spaces, BOM characters, and other invisible Unicode that some PDFs embed
  between words. These were preventing entity name matching even after
  whitespace normalization. Non-breaking spaces are replaced with regular
  spaces (not deleted) to preserve word boundaries.
- Added shorter entity alias "Certified Building Inspection" (4 words) as
  fallback — less vulnerable to text extraction splitting across 5 words.
  Also added full URL form "www.certifiedbuildinginspection.com.au".


v1.5.1 - 18 Mar 2026
---------------------
FIXED
- Entity detection now normalises whitespace before matching: PDF line breaks
  within multi-word entity names (e.g. "Certified Building Inspection\nServices")
  no longer prevent detection. This was the root cause of the Pre Purchase
  Inspection Report entity being blank despite the name appearing on page 1.
- TB32 Technical Bulletin now always shows ENTITY = BlueScope (forced override)
- Added BlueScope as a recognised FF entity with aliases (BlueScope Steel,
  BlueScope Steel Limited)


v1.5.0 - 18 Mar 2026
---------------------
FIXED
- Letter of Engagement now always shows ENTITY = ClaimsCo: forced entity
  overrides moved AFTER the entity_include_rules check so they can never
  be cleared by settings
- Request for Information now reliably shows insurer entity (e.g. AAI):
  settings migration ensures entity_include_rules is updated from old
  saved settings
- Entity detection now scans up to 30 pages (was 5) when no entity is
  found in the first 10 pages, catching entities deeper in long documents

IMPROVED
- Added settings migration system: when the app upgrades, new doc types,
  entities, aliases, and rule changes are automatically merged into the
  user's saved settings.json without losing custom modifications
- Increased default text extraction from 5 to 10 pages for better date,
  keyword, and entity detection across longer documents
